## 1. 분리 원칙

**200줄 제한은 안전망이지 목표가 아니다. 분리 기준은 책임이다.**
파일을 나눌지 판단할 때 이 세 가지를 묻는다:

1. **이 파일을 수정하는 이유가 하나인가?** — 두 가지 이유로 수정된다면 두 파일이다
   (예: "폼 검증 규칙이 바뀌어서"와 "폼 디자인이 바뀌어서" → schema와 컴포넌트 분리)
2. **파일 이름만 보고 무엇이 바뀔지 알 수 있는가?** — `hero.tsx`를 고치면 히어로만 바뀌어야 한다
3. **두 페이지 이상에서 쓰이는가?** — 그렇다면 `common/`, 한 페이지 전용이면 `components/<page>/`

### 계층별 책임 (하면 안 되는 것까지 명시)

| 계층 | 위치 | 책임 | 금지 |
|---|---|---|---|
| 라우트 | `app/*/page.tsx` | metadata 선언, 섹션 조립 순서 | 마크업·스타일·상태 소유 |
| 페이지 섹션 | `components/<page>/` | 한 섹션의 마크업+스타일 | 다른 섹션 참조, 데이터 값 정의 |
| 공통 조각 | `components/common/` | 재사용 프레젠테이션 | 특정 페이지 지식, content 직접 import |
| 사이트 골격 | `components/layout/` | 헤더/푸터/플로팅 (전 페이지 공통 프레임) | 페이지별 분기 로직 |
| 데이터 | `content/`, `config/` | 값 정의 | 렌더링, 스타일 |
| 디자인 값 | `styles/` | 토큰 정의 | 컴포넌트 |
| 순수 로직 | `lib/` | 스키마·유틸·레지스트리 | UI |
| API | `app/api/` | 요청 검증 + 전달 | 검증 규칙 정의(lib의 스키마를 공유) |

### 데이터 흐름 규칙 (누가 content를 import하는가)

- **페이지 섹션**은 자기 데이터(`content/*`)를 **직접 import**한다. 페이지가 props로 내려주지 않는다
  → 섹션과 데이터가 1:1이므로 중간 전달은 낭비이고, 페이지는 조립만 남는다
- **공통 조각**(Button, Section, Card)은 콘텐츠를 **props로만** 받는다. 스스로 import하지 않는다
  → 어떤 페이지에서도 재사용 가능해야 하므로
- **`config/site.ts`(연락처류)**는 예외적으로 어느 계층이든 import 가능 (전역 사실이므로)

### 클라이언트 경계

- styled-components를 쓰는 파일은 전부 `"use client"` — 즉 **섹션/조각/골격은 모두 클라이언트**
- `page.tsx`는 서버 컴포넌트 유지 (metadata 담당)
- **상태(useState)는 그것을 쓰는 leaf에만** 둔다: 모바일 메뉴 상태는 `mobile-menu.tsx` 안에,
  폼 상태는 `consult-form.tsx` 안에. 상위로 끌어올리지 않는다
- FAQ는 상태 없이 네이티브 `<details>`로 구현 (JS 불필요)

### 스타일 정의 위치

- styled 정의는 **그것을 쓰는 컴포넌트 파일 안에** 둔다 (같이 바뀌므로 같은 파일)
- 스타일 분량이 파일의 절반을 넘어 컴포넌트 로직이 안 보이면 `*.styles.ts`로 분리 — 그 전엔 분리하지 않는다

## 2. F1 — 공통 컴포넌트 (이번 단계)

### `components/common/`

| 파일 | 책임 한 줄 | 주요 인터페이스 |
|---|---|---|
| `button.tsx` | 링크형/버튼형 버튼의 모양 | `ButtonLink`, `Button` — `$variant`: primary·accent·outline·outlineOnDark, `$size` |
| `section.tsx` | 섹션 골격(배경 변형 + 폭 + 여백 + 헤딩) | `Section($bg)`, `SectionInner($narrow)`, `SectionHeading(label,title,description,onDark)` |
| `card.tsx` | 카드 프레임(보더+라운드+배경)의 모양 | `Card` styled 조각 |
| `cta-banner.tsx` | "상담이 필요하신가요" 배너 (4개 페이지 공통 사용) | props 없음 — `content/cta.ts` + `SITE` 소비 |
| `page-hero.tsx` | 서브페이지 상단 히어로(라벨+h1+설명) | props로 label/title/description (F3에서 첫 사용 시 구현) |

`cta-banner`는 공통 조각이지만 props가 아니라 content를 직접 소비한다 — 사이트에 단 하나뿐인
고정 콘텐츠 섹션이기 때문. "여러 곳에서 같은 내용으로 재사용"은 섹션의 성격을 따른다.

### `components/layout/`

| 파일 | 책임 한 줄 | 비고 |
|---|---|---|
| `header.tsx` | 스티키 헤더의 레이아웃(로고+데스크톱 GNB+CTA) | 활성 메뉴 표시(usePathname). 모바일 메뉴는 소유하지 않고 배치만 |
| `mobile-menu.tsx` | 햄버거 버튼과 드로어의 열림/닫힘 | 상태는 이 파일에만 존재 |
| `footer.tsx` | 푸터 3열 + 하단 저작권 바 | `nav` + `SITE` 소비 |
| `floating-actions.tsx` | 빠른 연락 진입점(데스크톱 플로팅 + 모바일 하단 바) | 같은 책임의 뷰포트별 두 형태이므로 한 파일 |

### 데이터/토큰 추가

| 파일 | 추가 내용 |
|---|---|
| `content/cta.ts` | CTA 배너 문구 (제목, 설명, 버튼 라벨) |
| `config/site.ts` | `disclaimer`(예시 고지 문구) 추가 |
| `styles/theme.ts` | `layout.mobileBarH`(모바일 하단 바 높이) 추가 |
| `styles/media.ts` | `maxMd`(모바일 전용 스타일용) 추가 |
| `app/layout.tsx` | Header / children / Footer / FloatingActions 배치 |

로고 이미지는 아직 에셋이 없으므로 `SITE.name` 첫 글자로 만든 스타일 마크로 대체한다
(에셋 수령 시 `header.tsx`의 마크 부분만 교체).

## 4. Feature별 완료 기준 (공통)

1. `npm run check:tokens` 통과 (하드코딩 0)
2. `npm run build` 통과 (타입 에러 0)
3. 실제 렌더링 확인 (서버 기동 → 해당 페이지 응답에 콘텐츠·스타일 존재)
4. 모든 파일 200줄 미만 — 넘으면 줄이기가 아니라 **책임 분리**로 해결
5. 커밋 단위 = feature 하나 (커밋 메시지 한국어, feat/fix/refactor 접두사)
