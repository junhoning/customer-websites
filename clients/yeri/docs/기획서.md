- 작성일: 2026-07-15

## 1. 핵심 원칙: 단일 소스(Single Source of Truth)

> **모든 디자인 값·사이트 정보·콘텐츠는 토큰/설정 파일에서만 정의하고,
> 페이지·컴포넌트는 100% 여기서 불러와서 사용한다. 예외 없음.**

수정이 필요할 때 건드리는 파일은 아래 3곳뿐이다:

| 바꾸고 싶은 것 | 수정할 파일 | 예시 |
|---|---|---|
| 색·폰트·둥글기·간격 등 **테마** | `styles/theme.ts` | 네이비 → 그린 테마 전환 |
| 전화번호·주소·사무소명 등 **사이트 정보** | `config/site.ts` | 실제 연락처 입력 |
| 서비스·FAQ·수치 등 **콘텐츠** | `content/*.ts` | FAQ 항목 추가 |

컴포넌트 코드(`app/`, `components/`)에는 hex 색상, 폰트명, 전화번호·주소·문구 리터럴이
**한 글자도 들어가지 않는다.**

## 2. 토큰 아키텍처 (3계층)

```
[1층] Primitive 팔레트  — 원시 값(색상 스케일). theme.ts 내부에만 존재, export 금지
        ↓ 참조
[2층] Semantic 테마     — 역할 이름. primary, accent, surface… 유일하게 export되는 객체
        ↓ ThemeProvider로 주입
[3층] 컴포넌트/페이지    — ${({ theme }) => theme.color.primary} 형태로만 사용
```

**강제 방식이 핵심이다:**
- 1층 팔레트는 `theme.ts` 안의 비공개 상수 → **컴포넌트가 primitive를 import하는 것 자체가 불가능**
- 2층은 `DefaultTheme` 타입으로 선언 → 오타·없는 토큰 사용 시 **컴파일 에러**
- 색이 하나 필요해지면 컴포넌트에 쓰는 게 아니라 theme.ts에 semantic 토큰을 추가한다

## 3. 디자인 토큰 정의 — `styles/theme.ts`

**이 파일이 테마의 전부다.** 레퍼런스 사이트에서 추출한 실제 값 기준.

```ts
/* ========== [1층] Primitive — export 하지 않는다 ========== */
const palette = {
  navy50:  "#eef3fa",
  navy600: "#1e498a",
  navy700: "#16386a",
  navy800: "#10294c",
  navy900: "#0b1f3a",

  gold100: "#f6edd8",
  gold500: "#c8a34e",
  gold600: "#b8912f",
  gold700: "#846818",

  red400: "#ff6568",
  red600: "#e40014",

  white: "#ffffff",
} as const;

/* ========== [2층] Semantic — 컴포넌트가 사용하는 유일한 이름 ========== */
export const theme = {
  color: {
    /* 배경/텍스트 기본 */
    background: palette.white,     // 페이지 기본 배경
    foreground: "#1a2233",         // 본문 텍스트
    muted:      "#5b6577",         // 보조 텍스트
    surface:    "#f7f9fc",         // 섹션 교차 배경
    line:       "#e4e8f0",         // 보더/구분선

    /* 브랜드 역할 */
    primary:       palette.navy700,  // 주 버튼 배경
    primaryStrong: palette.navy800,  // 어두운 배너, 헤딩 텍스트
    primaryDeep:   palette.navy900,  // 히어로 그라디언트 시작
    primaryHover:  palette.navy600,  // hover, focus 보더
    primaryTint:   palette.navy50,   // 밝은 칩/배지 배경

    accent:      palette.gold500,    // 골드 포인트(버튼, 장식선)
    accentHover: palette.gold600,
    accentText:  palette.gold700,    // 라벨, 필수(*) 표시
    accentTint:  palette.gold100,

    /* 상태 */
    danger:       palette.red600,    // 폼 에러 메시지
    dangerBorder: palette.red400,    // 에러 인풋 보더

    /* 외부 브랜드 고정색 (테마와 무관) */
    kakao:     "#fee500",
    kakaoText: "#3c1e1e",

    onDark: palette.white,           // 어두운 배경 위 텍스트
  },

  font: {
    heading: `var(--font-nanum), "Nanum Myeongjo", serif`,      // 제목: 나눔명조
    body:    `var(--font-pretendard), -apple-system, sans-serif`, // 본문: Pretendard
  },

  fontSize: {
    xs: "0.75rem", sm: "0.875rem", base: "1rem", lg: "1.125rem",
    xl: "1.25rem", h3: "1.5rem", h2: "1.875rem", h1: "2.25rem",
    hero: "2.75rem",               // 히어로 h1 (md 이상)
  },

  radius: {
    card:  "0.75rem",   // 카드
    field: "0.375rem",  // 인풋/버튼
    pill:  "9999px",    // 배지/원형 버튼
  },

  layout: {
    content:  "72rem",   // 콘텐츠 최대폭
    narrow:   "48rem",   // 좁은 텍스트 섹션
    sectionY: "5rem",    // 섹션 수직 여백
    gutter:   "1.25rem", // 좌우 패딩
    headerH:  "4.5rem",  // 헤더 높이 (scroll-margin 보정에도 사용)
  },

  z: { floating: 30, header: 40, mobileBar: 40, modal: 50 },
} as const;

export type AppTheme = typeof theme;
```

## 4. Next.js App Router 연동 설정

styled-components를 App Router에서 쓰기 위한 필수 설정. **F0에서 가장 먼저 구축한다.**

1. `next.config.ts`: `compiler: { styledComponents: true }` (SSR 클래스명 일치 + displayName)
2. `lib/registry.tsx`: `StyledComponentsRegistry` — `ServerStyleSheet`로 SSR 스타일 주입
   (Next 공식 문서의 styled-components 레지스트리 패턴 그대로)
3. `app/layout.tsx`(서버 컴포넌트): next/font로 나눔명조·Pretendard 로드 →
   `--font-nanum`, `--font-pretendard` CSS 변수 주입 → Registry + Providers로 children 감싸기
4. `app/providers.tsx`(`"use client"`): `ThemeProvider theme={theme}` + `<GlobalStyle />`

**규칙**: styled-components를 쓰는 파일은 전부 `"use client"`.
`page.tsx`는 서버 컴포넌트로 유지(메타데이터 export)하고, 화면은 클라이언트 섹션
컴포넌트를 조립하는 방식으로 구성한다.

```
app/page.tsx (서버: metadata + 데이터 전달)
  └─ components/home/hero.tsx ("use client": styled + 렌더링)
```

## 6. 콘텐츠 토큰 — `content/*.ts`

페이지에 뿌려지는 모든 데이터. **문구 수정 = 이 폴더만 수정.**

| 파일 | 내용 | 사용처 |
|---|---|---|
| `content/services.ts` | 6대 분야: `{ id, name, tagline, description, steps[5], documents[3] }` | 홈 카드, 서비스 상세, 폼 select, 앵커 (id 공유) |
| `content/stats.ts` | 실적 수치 4개: `{ value, label }` | 홈 수치 밴드 |
| `content/process.ts` | 진행 절차 5단계: `{ step, title, description }` | 홈 절차 섹션 |
| `content/faq.ts` | FAQ 6개: `{ question, answer }` | 홈 FAQ |
| `content/profile.ts` | 대표 소개 문단, 경력 불릿 4개, 강점 카드 3개 | about |
| `content/nav.ts` | GNB/푸터 메뉴: `{ label, href }` | 헤더, 푸터 |
| `content/privacy.ts` | 개인정보처리방침 8개 조항 | privacy |

`services.ts`의 `id`("patent", "trademark"…)가 **앵커 링크 · 폼 select 값 · Zod enum**을
전부 구동한다. 분야 추가/삭제 시 이 배열 하나만 수정하면 사이트 전체에 반영된다.

```ts
// content/services.ts 형태
export const SERVICES = [
  {
    id: "patent",
    name: "특허",
    tagline: "기술 아이디어를 독점권으로. 명세서 작성부터 등록까지.",
    description: "발명의 핵심을 정확히 포착한 명세서 설계로…",
    steps: ["상담 및 발명 상담", "선행기술 조사", "명세서 작성·출원", "심사·의견제출통지 대응", "등록 결정"],
    documents: ["발명 설명 자료", "도면 또는 스케치", "사업자·출원인 정보"],

export type ServiceId = (typeof SERVICES)[number]["id"];
```

## 7. 100% 토큰 사용 강제 수단

### 7.1 타입 시스템 (1차 방어 — 가장 강력)

- primitive 팔레트는 export하지 않으므로 **컴포넌트에서 import 자체가 불가능**
- `theme.color.primry` 같은 오타 → **컴파일 에러** (DefaultTheme 타입)
- 새 색 필요 → theme.ts에 semantic 토큰 추가 후 사용하는 것 외에 방법이 없음

### 7.2 검사 스크립트 (2차 방어) — `scripts/check-tokens.sh`

타입으로 못 막는 "리터럴 직접 입력"을 탐지한다. 발견 시 exit 1.

```bash
#!/usr/bin/env bash
FAIL=0

# 1) hex/rgb 색상 리터럴 (styles/theme.ts 제외)
if grep -rEn '#[0-9a-fA-F]{3,8}\b|rgba?\(' app components --include='*.tsx' --include='*.ts'; then
  echo "❌ 색상 리터럴 발견 — styles/theme.ts에 토큰으로 정의하세요"; FAIL=1
fi

# 2) 인라인 style 색상
if grep -rEn 'style=\{\{[^}]*(color|background)' app components --include='*.tsx'; then
  echo "❌ 인라인 스타일 색상 발견"; FAIL=1
fi

# 3) 폰트명 리터럴
if grep -rEn 'Nanum|Pretendard|Myeongjo' app components --include='*.tsx'; then
  echo "❌ 폰트명 하드코딩 — theme.font를 쓰세요"; FAIL=1
fi

# 4) 사이트 정보 리터럴 (전화번호, 예시 이메일, 주소)
if grep -rEn '0[0-9]{1,2}-[0-9]{3,4}-[0-9]{4}|@example\.com|서초구' app components --include='*.tsx'; then
  echo "❌ 연락처/주소 리터럴 발견 — config/site.ts에서 import 하세요"; FAIL=1
fi

# 5) @media 직접 작성 (styles/media.ts 외)
if grep -rEn '@media' app components --include='*.tsx'; then
  echo "❌ @media 직접 작성 — styles/media.ts의 media 토큰을 쓰세요"; FAIL=1
fi

exit $FAIL
```

### 7.3 실행 시점

- `package.json`: `"check:tokens": "bash scripts/check-tokens.sh"`, `prebuild`에 연결
- 기능 구현 각 Phase 완료 시 실행 → 통과가 완료 조건

### 7.4 코드리뷰 체크리스트

- [ ] styled 블록 안에 리터럴 값(색·폰트·max-width 등)이 없는가 — 전부 `${({ theme }) => …}`인가
- [ ] 새 색이 필요했다면 theme.ts에 semantic 토큰을 추가했는가
- [ ] 새 문구/데이터는 `content/`에, 연락처류는 `config/site.ts`에 있는가
- [ ] `@media` 대신 `media.md` 토큰을 썼는가

## 9. 페이지별 구현 계획

각 페이지가 어떤 토큰을 소비하는지 명시한다. 상세 섹션 구성은 분석 문서 5장 참고.

| 페이지 | 사용하는 콘텐츠 토큰 | 주요 semantic 토큰 |
|---|---|---|
| 홈 `/` | services, stats, process, faq + SITE | primaryDeep/Strong(히어로 그라디언트), surface(수치 밴드), accent(포인트) |
| `/about` | profile + SITE(오시는 길) | surface, primaryTint, line |
| `/services` | services (전체 필드) | primaryTint(앵커 탭), line(카드) |
| `/contact` | services(select 옵션) + SITE(채널 안내) | danger/dangerBorder(에러), primary(제출 버튼) |
| `/privacy` | privacy + SITE(책임자 연락처) | muted, line |
| 공통 | nav + SITE | header: background 반투명+blur, footer: surface, CTA배너: primaryStrong |

### 구현 순서 (feature 단위, 한 번에 하나씩)

1. **F0 — 기반 구축**: Next 프로젝트 생성 → styled-components 설정(registry, compiler 옵션)
   → `styles/`(theme·media·global·styled.d.ts) → `config/site.ts` → `content/*` →
   `check-tokens.sh` → layout(폰트/Providers) — 검사 스크립트 통과 확인
2. **F1 — 공통 컴포넌트**: Button/Section/Card 등 공용 조각 → Header(+모바일 메뉴),
   Footer, CTA배너, 플로팅 액션(데스크톱 우하단 + 모바일 하단 바)
3. **F2 — 홈**: 히어로 → 수치 밴드 → 분야 카드 → 절차 → FAQ(`<details>`) → CTA
4. **F3 — 서비스**: 앵커 탭 + SERVICES map 렌더링
5. **F4 — 소개**: 프로필, 강점 카드, 오시는 길(지도는 플레이스홀더 유지, 연동 여부 발주자 확인)
6. **F5 — 상담**: ConsultForm(Zod) + `/api/consult` Route Handler + 접수 처리
   (수신 방식 발주자 확인: 이메일/슬랙/DB)
7. **F6 — 개인정보처리방침 + SEO**: privacy 페이지, metadata, sitemap/robots, OG 이미지

각 feature 완료 조건: 동작 확인 + `npm run check:tokens` 통과 + 파일 200줄 미만.
