- 관련: `clients/*/feedback/` (수집 결과의 최종 목적지 — 엑셀 파이프라인)

> 이 문서에서 §2 원칙 3(무의존성·30KB), §5.2 데이터 모델, §5.4 작성 폼 UX(유형·중요도),

## 1. 무엇을 만드는가

리뷰용 배포에 주입하는 **페이지 내 피드백 수집 위젯**.

- 고객이 리뷰 링크로 접속 → 피드백 모드를 켜고 → 화면의 특정 요소에 **우클릭(모바일: 롱프레스)** → 그 자리에서 피드백 작성
- 우측 사이드바에 피드백 목록이 쌓이고, 항목을 누르면 **작성했던 위치로 바로 점프** + 하이라이트
- 결과는 기존 엑셀 파이프라인(`feedback_data.py` → xlsx)으로 합류한다

**해결하는 문제**: "중간쯤 파란 부분이요" — 위치 특정에 드는 왕복 비용을 0으로 만든다.

## 2. 원칙

1. **고객사 코드 무오염** — 고객사에 들어가는 것은 layout의 조건부 `<Script>` 한 줄뿐.
   환경변수가 꺼진 빌드(실제 오픈)에는 위젯 코드가 0바이트도 포함되지 않는다
2. **스타일 격리** — Shadow DOM 안에서만 렌더링. 고객사 CSS와 위젯 CSS는 서로 영향 없음
3. **무의존성** — 프레임워크 없이 바닐라 TS, esbuild로 단일 IIFE(`widget.js`) 빌드.
   어떤 스택의 고객사에도 붙는다. 목표 크기 30KB 이하
4. **엑셀과 같은 어휘** — 유형·중요도·진행상태 값을 엑셀 양식과 동일하게 사용
5. **v0는 서버 없음** — localStorage + JSON 내보내기로 UX부터 검증. 서버(hub)는 v1

## 3. 환경 모델

| 환경 | 위젯 | 링크 |
|---|---|---|
| 로컬 개발 | `.env.local`로 선택 | localhost:3000 (본인 확인용) |
| **리뷰 배포** | **포함** | **고객에게 전달** (예: `yeri-review.vercel.app?fb=토큰`) |
| 실제 오픈 | 미포함 | 고객 도메인 |

고객사 통합(전 고객 공통, 이 형태 그대로):

```tsx
{process.env.NEXT_PUBLIC_FEEDBACK === "1" && (
  <Script src={process.env.NEXT_PUBLIC_FEEDBACK_URL} data-project="yeri" />
)}
```

## 4. v0 범위 (이번 구현)

**포함**: 토글 버튼 / 우클릭·롱프레스 캡처 / 작성 폼 / 사이드바 목록 / 위치 점프+하이라이트 /
핀 마커 / localStorage 저장 / JSON 내보내기(다운로드+클립보드) / 병합 스크립트(JSON→feedback_data.py) /
**메타데이터 자동 수집**(userAgent·화면 크기 — 상용 도구 벤치마크. "오류 신고 시 기기를 적어달라"를 자동화)

**제외 (v1+)**: 수집 서버(hub)·관리자 뷰 / 접근 토큰 검증 / 스크린샷 / 답글 스레드 /
상태 실시간 동기화 / 알림. — v0에서 상태(진행 상태)는 **표시만** 하고 변경은 작업자가 엑셀에서 관리

**v0의 한계 (의도된 것)**: 피드백이 고객 브라우저에만 저장되므로, 고객이 "내보내기"를 눌러
파일을 카톡/메일로 보내야 우리에게 도달한다. 이 마찰이 크면 그때 v1(hub)로 올린다.

## 5. 핵심 설계

### 5.1 위치 앵커 — 다중 신호 저장, 우선순위 복원

```ts
interface Anchor {
  page: string;        // pathname ("/services")
  selector: string;    // CSS 경로 ("section#patent > h2")
  textSnippet: string; // 요소 텍스트 앞 40자 (셀렉터 깨졌을 때 폴백)
  scrollY: number;     // 최후 폴백
}
```

복원: `querySelector` → 실패 시 textSnippet 포함 요소 탐색 → 실패 시 scrollY 이동.
성공 시 요소에 2초간 하이라이트 아웃라인. 다른 페이지 항목은 sessionStorage에
"점프 예약"을 남기고 이동 → 로드 후 위젯이 예약을 소비한다.
셀렉터 생성 시 id 우선, 없으면 `태그:nth-of-type` 경로. 위젯 자신의 요소는 캡처 대상에서 제외.

### 5.2 데이터 모델 (엑셀과 어휘 통일)

```ts
interface FeedbackItem {
  id: string;              // crypto.randomUUID()
  createdAt: string;       // ISO
  author: string;          // 선택 입력, 기본 "고객"
  anchor: Anchor;
  type: "문구" | "디자인" | "구조" | "정보" | "기능" | "오류";
  priority: "A" | "B" | "C";
  content: string;
  status: "대기" | "진행 중" | "완료" | "확인 필요";  // v0: 표시 전용
  meta: { userAgent: string; viewport: string };       // 자동 수집
}
```

- localStorage 키: `fbw:<project>` (예: `fbw:yeri`)
- 내보내기 JSON: `{ project, exportedAt, items: FeedbackItem[] }`

### 5.3 엑셀 파이프라인 합류

`tools/merge_feedback_json.py <json파일> clients/<고객>`:
JSON 항목을 `feedback_data.py`의 FEEDBACK_ITEMS 형식 튜플로 변환해 **append**
(위치 열은 `pathname` 기반 자동 표기 + anchor 정보는 참고 자료 열에 기록,
섹션 이름 매핑은 작업자가 엑셀에서 보정). 변환 후 기존 생성기로 xlsx 재생성.

### 5.4 UX 명세

- **토글**: 좌하단 플로팅 버튼(고객사 플로팅 버튼과 겹치지 않게 좌측). OFF면 위젯은
  아무 이벤트도 가로채지 않는다. ON이면 배지에 항목 수 표시
- **캡처**: ON 상태에서 hover 시 요소에 점선 아웃라인(개발자도구 느낌).
  우클릭(데스크톱) 또는 500ms 롱프레스(모바일) → 작성 폼 팝오버
- **작성 폼**: 내용(필수) / 유형(기본: 문구) / 중요도(기본: B) / 작성자(선택, 기억됨).
  저장하면 그 위치에 번호 핀 표시
- **사이드바**: 우측 슬라이드. 목록(번호·유형·내용 요약·상태 뱃지), 클릭=점프,
  삭제(본인 작성분), 하단에 "내보내기" 버튼 2개(JSON 다운로드 / 클립보드 복사)
- **핀**: 현재 페이지의 앵커 위치에 번호 뱃지 오버레이. 스크롤 따라 이동(absolute 배치)

### 5.5 위젯 내부 토큰

위젯 UI 색·크기도 `src/tokens.ts` 한 파일에 상수로 모은다 (프로젝트 전체 원칙과 동일).
고객사 테마와 무관한 중립 색(진회색+화이트)을 쓴다 — 어떤 사이트 위에서도 이질감 최소화.

## 6. 파일 구조와 책임 (파일당 200줄 미만)

```
common/feedback-widget/
├── docs/기획서.md            # 이 문서
├── widget/
│   ├── package.json          # esbuild, typescript (자체 완결)
│   ├── tsconfig.json
│   ├── src/
│   │   ├── index.ts          # 부트스트랩: 설정 읽기, Shadow DOM 마운트, 모듈 조립만
│   │   ├── tokens.ts         # 위젯 UI 토큰 (색·크기·z-index)
│   │   ├── store.ts          # 항목 CRUD + localStorage 영속화 (UI 모름)
│   │   ├── anchor.ts         # 위치 캡처/복원/하이라이트 (순수 DOM 로직)
│   │   ├── export.ts         # JSON 직렬화·다운로드·클립보드
│   │   └── ui/
│   │       ├── toggle.ts     # 플로팅 토글 버튼
│   │       ├── capture.ts    # hover 아웃라인 + 우클릭/롱프레스 이벤트
│   │       ├── form.ts       # 작성 폼 팝오버
│   │       ├── sidebar.ts    # 목록·점프·삭제·내보내기
│   │       └── pins.ts       # 번호 핀 오버레이
│   └── dist/widget.js        # 빌드 산출물 (커밋함 — 고객사가 바로 로드)
└── tools/
    └── merge_feedback_json.py  # JSON → clients/<고객>/feedback/feedback_data.py 병합
```

- UI 모듈은 store를 구독하고, store는 UI를 모른다 (단방향)
- anchor/store/export는 DOM 이벤트를 직접 걸지 않는다 — 이벤트는 ui/에서만

## 7. 검증 기준 (v0 완료 조건)

1. `widget/npm run build` → 단일 `dist/widget.js`, 30KB 이하, 외부 요청 0건
2. yeri 로컬(`NEXT_PUBLIC_FEEDBACK=1`)에서: 작성→새로고침 후 유지→다른 페이지
   항목 클릭 시 페이지 이동+점프+하이라이트 동작
3. 사이트를 일부 수정해 셀렉터를 깨뜨려도 텍스트 폴백으로 점프 성공
4. 내보낸 JSON을 `merge_feedback_json.py`로 병합 → xlsx 재생성까지 무오류
5. 위젯 ON 상태에서 고객사 기능(폼 제출, 아코디언, 메뉴) 정상 동작,
   OFF 상태에서 우클릭 등 브라우저 기본 동작 정상
6. 모든 파일 200줄 미만, `NEXT_PUBLIC_FEEDBACK` 미설정 프로덕션 빌드에 위젯 흔적 없음

## 8. 호스팅 방향 (Firebase 확정) 및 미결정 사항

호스팅은 **Firebase**로 간다 (도메인 연결·확장성·무료 티어).

| 항목 | v0 임시안 | 방향 |
|---|---|---|
| 수집 서버 저장소 | — (localStorage) | v1: **Firestore** |
| 접근 제어 | 링크 비공개에 의존 | v1: `?fb=토큰` 검증 (Firestore 보안 규칙 병행) |
| widget.js 호스팅 | yeri public/에 복사(임시) | **Firebase Hosting** (CDN, 전 고객 공용) |
| 리뷰 배포 방식 | 미정 | **Firebase Hosting preview channel** — 만료 기간 있는 토큰 URL을 고객 리뷰 링크로 사용 |

**yeri 배포 시 결정할 것**: Next.js를 ① Firebase App Hosting(SSR — `/api/consult` 유지)으로
올릴지 ② 정적 내보내기 + 클래식 Hosting(상담 접수는 Cloud Functions로 분리)으로 갈지.
사이트는 상담 API 하나 빼면 전부 정적이므로 둘 다 가능 — 비용은 ②가 유리, 코드 유지 관점은 ①이 단순.
