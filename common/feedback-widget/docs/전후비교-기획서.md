# 전후 비교 기획서 — feedback-widget v0.6 (dev 모드)

> 작성일 2026-07-16. [comment-전환-기획서.md](comment-전환-기획서.md) §10-3의 구체화.
> **전제: 당분간 Firebase에 올리지 않는다.** 서버 없이 dev 모드 + git만으로
> "코멘트 작성 시점의 화면"과 "작업 반영 후 화면"을 나란히 비교한다.

## 1. 목표

고객이 코멘트 스레드에서 **[Compare]** 버튼을 누르면, 그 코멘트를 **작성했던 시점의
화면(Before)**과 **현재 화면(After)**이 나란히 뜨고, 두 화면 모두 해당 코멘트의
앵커 위치로 스크롤되어 "무엇이 어떻게 바뀌었는지"를 눈으로 확인한다.

핵심 통찰: 비교 뷰가 실제로 요구하는 것은 **URL 두 개**뿐이다.
git이 모든 버전을 보관하므로, Before는 "그 시점 커밋을 임시로 띄운 서버"면 된다.
나중에 Firebase preview channel이 생기면 **Before URL만 교체**되고 나머지는 재사용된다.

## 2. 구성 요소 (3개)

```
[1] 버전 기록      스레드에 "작성 시점 커밋 해시" 한 줄을 기록
[2] Before 서버    그 커밋을 git worktree로 꺼내 :3001에 띄우는 스크립트 (작업자용)
[3] 비교 뷰        위젯 안의 전체 화면 오버레이 — iframe 2개 + 앵커 동기 스크롤
```

### 2.1 버전 기록 — 스레드 스키마에 `version` 추가

- yeri `package.json`의 dev/build 스크립트가 `NEXT_PUBLIC_GIT_SHA=$(git rev-parse
  --short HEAD)`를 주입 → layout의 위젯 `<Script>`에 `data-version`으로 전달
- 위젯은 새 스레드 저장 시 `version: <sha>`를 함께 기록
- **additive 변경** — 필드 추가뿐이므로 스키마 키 승격(v3) 불필요, 마이그레이션 없음.
  `version` 없는 기존 스레드는 Compare 버튼이 비활성(이유 툴팁)
- 한계(수용): dev 서버는 시작 시점의 해시로 고정된다. 서버를 켠 채 커밋하면
  이후 스레드에 이전 해시가 적힌다 — 리뷰 배포 단위로 서버를 재시작하는 운영으로 커버

### 2.2 Before 서버 — `tools/compare-serve.sh` (작업자용)

```
사용법: common/feedback-widget/tools/compare-serve.sh <고객폴더> <커밋해시> [포트=3001]
동작:  git worktree add (임시 폴더, 해당 해시)
       → 고객 폴더의 .env.local 복사 (gitignore라 worktree에 없음 — 위젯 활성화에 필요)
       → npm install → dev 서버를 -p <포트> -H 0.0.0.0 으로 실행
종료:  Ctrl+C 시 worktree 정리 (git worktree remove)
```

- Before 페이지에도 위젯이 실려야 한다(§2.3의 동기 스크롤 수신부) — .env.local 복사 이유
- 고객 시연: 지금 3000을 포워딩하듯 3001도 열면 원격에서도 비교 가능

### 2.3 비교 뷰 — 위젯 오버레이

- ThreadView에 **[Compare]** 버튼 추가. 활성 조건: `thread.version` 존재
- 클릭 → 전체 화면 오버레이(Shadow DOM 안):
  - 헤더: `Before <sha>` | `After <sha(현재)>` | Close
  - 본문: iframe 2개 좌우 분할 (모바일: 상하)
    - Before: `<compareBase>` + 스레드의 anchor.page
    - After: `location.origin` + anchor.page
- **compareBase 결정**: 기본값 `${location.protocol}//${location.hostname}:3001`
  — 고객이 어떤 호스트로 접속했든 같은 호스트의 3001을 본다.
  위젯 `<Script data-compare-base>`로 재정의 가능 (Firebase 전환 시 이 값만 바뀐다)
- **앵커 동기 스크롤 — postMessage**: 3000과 3001은 서로 다른 origin이라
  부모가 iframe 내부를 직접 스크롤할 수 없다. 대신 양쪽 페이지에 실린 위젯이 수신부가 된다:
  1. iframe URL에 `?fbw=embed`를 붙인다 → 위젯이 **embed 모드**로 부팅
     (UI 전부 숨김, message 리스너만 활성)
  2. 부모 위젯이 두 iframe에 `postMessage({ type: "fbw:jump", anchor })` 전송
  3. embed 위젯이 기존 `jump()`(셀렉터 → 텍스트 폴백 → scrollY)로 이동 + 하이라이트
  - Before 시점에는 요소가 없거나 다를 수 있다 — 다중 신호 앵커의 폴백이 그대로 안전망

## 3. UX 흐름 (전체)

```
고객: 코멘트 작성                          → 스레드에 version: abc1234 기록
작업자: 수정 반영, 커밋(def5678), 서버 재시작
작업자: compare-serve.sh clients/yeri abc1234   → :3001에 Before 서버
고객: 스레드 열기 → [Compare] → Before(abc1234) | After(def5678) 나란히,
      해당 위치로 양쪽 모두 스크롤 + 하이라이트 → 확인 후 답글 or Archive
```

## 5. 검증 기준 (v0.6 완료 조건)

1. 새 스레드에 version이 기록되고, version 없는 스레드는 Compare 비활성
2. `?fbw=embed`로 부팅 시 위젯 UI가 일절 보이지 않고, fbw:jump 메시지에 점프한다 (smoke)
3. Compare 오버레이에 iframe 2개가 올바른 URL(3001/현재 origin + anchor.page)로 뜬다 (smoke)
4. 실사용: 과거 커밋을 compare-serve.sh로 띄워 테마 변경 전후(#16386a 남색 ↔ 차콜)를
   나란히 확인 — 피드백 1번이 살아 있는 검증 사례
5. embed 모드가 아닌 일반 부팅·기존 테스트 14종에 회귀 없음
6. 모든 파일 200줄 미만

## 6. Firebase 전환 시 (나중)

| 항목 | 지금 (dev) | Firebase 이후 |
|---|---|---|
| Before URL | 같은 호스트 :3001 (수동 서빙) | preview channel URL (배포마다 자동) |
| version 기록 | git SHA | git SHA (동일) — 채널 URL과 매핑만 추가 |
| 비교 뷰·동기 스크롤 | 이번에 구현 | **그대로 재사용** |
| 작업자 수고 | 비교할 때마다 스크립트 실행 | 없음 (URL이 상시 존재) |
